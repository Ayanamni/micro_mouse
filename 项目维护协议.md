# 项目维护协议

本协议适用于 Codex、Claude，以及任何后续维护者。目标是让项目可以持续加模块、调控制器、换仿真模型，但不把系统堆成难以维护的状态。

## 1. 优先级

遇到冲突时按以下顺序处理：

1. 用户当前明确要求。
2. `项目维护协议.md`。
3. `AGENTS.md` / `CLAUDE.md`。
4. `控制器方案报告.md`。
5. `gpt计划.md` / `claude计划.md`。
6. 现有代码实现。

如果文档和代码冲突，不能默认代码正确；必须在回复或计划里写明冲突点，再决定修改文档还是修改代码。

## 2. 八条硬规则

### 规则 1：先契约，后代码

任何非平凡修改，先写清楚模块契约，再改代码。

契约至少包括：

- 输入、输出、单位。
- 调用频率。
- 状态所有权。
- 限幅/饱和/异常输入处理。
- debug 字段。
- 验收命令。

### 规则 2：区分 `cmd/ref/raw/alloc/duty`

控制链路必须使用固定术语：

| 名称 | 含义 |
|---|---|
| `cmd` | 上层命令，允许跳变和越界 |
| `ref` | 下层整形后的可执行参考，连续且导数有界 |
| `raw` | 控制器未分配/未最终限幅的请求 |
| `alloc` | 动态可行域分配后的实际力矩 |
| `duty` | 电机电压命令，范围 `[-1, 1]` |

禁止把 `cmd` 直接当作 `ref` 给前馈、DOB 或反馈。

### 规则 3：下层保护硬件，上层不可信

上层可以输出粗糙命令；下层控制器必须保护硬件和状态。

必须保证：

- 输入阶跃不产生无限加速度或角加速度。
- 错误大命令会被 clamp 或降额。
- 积分器、DOB、力矩分配不能因上层错误命令持续 windup。
- 电机命令永远满足电压、电流和 duty 限制。

### 规则 4：参数只能有一个默认源

控制器默认参数只能从一个共享默认源传播。

当前 Python 侧默认源是：

`micromouse_sim/config/vw_controller_config.py`

禁止在 `workbench.py`、`eval_vw.py`、`bench_vw.py` 里复制一套不同默认值。CLI override 可以存在，但默认值必须来自共享源。

### 规则 5：量具先于调参

调参前必须确认量具可信。

如果存在以下情况，必须先停下修量具：

- 指标读到的是旧值、raw 值或错误单位。
- step/profile 测试与设计假设冲突。
- 饱和统计不是实际执行值。
- debug 缺少判断问题所需字段。

没有可信量具时禁止继续靠感觉调参数。

### 规则 6：一次只解释一个原因

参数调优时，每轮只改一个主要变量，并记录改前/改后的指标。

结构性改造可以同时改多个文件，但必须保持一个目标，例如：

- “新增参考整形层”
- “把 yaw 输出从 1kHz 改成 5kHz”
- “暴露真实分配后的力矩”

不能在同一轮里混合结构改造、参数调优和巡线策略调整。

### 规则 7：主动路径必须写清楚

当前主动控制路径是：

`workbench/eval/bench -> control_core.vw_omega_step / vw_control_tick -> motor_model -> MuJoCo`

legacy `control_core.step()` 只允许做 A/B 对比，不能悄悄变成主路径。

任何修改必须说明影响哪些入口：

- `scripts/workbench.py`
- `scripts/eval_vw.py`
- `scripts/bench_vw.py`
- `scripts/test_omega_bandwidth.py`
- C++ pybind API

### 规则 8：代码改完必须验证

涉及代码后，至少运行：

```bash
python -m py_compile scripts\workbench.py scripts\eval_vw.py scripts\bench_vw.py
cmake --build cpp\build --config Release
```

涉及主动控制路径后，还必须运行：

```bash
python scripts\eval_vw.py single --track 2019kansai --speed 3.5 --duration 12 --seed 42
```

如果是角速度环或前馈/DOB修改，还要运行对应 bench/eval：

```bash
python scripts\bench_vw.py omega-step --mag 5
python scripts\eval_vw.py omega-step --track 2019kansai --speed 1.0 --omega-step-value 5.0
```

如果没有运行，最终回复必须明确说明未运行以及原因。

## 3. 新增模块流程

新增模块按这个流程，不跳步：

1. 写一段契约到相关计划或设计文档。
2. 明确模块放在哪里、谁调用、谁拥有状态。
3. 明确 debug 字段和日志字段。
4. 增加最小量具或复用已有量具。
5. 实现最小版本。
6. 运行验证命令。
7. 记录结果和下一步。

模块未完成时，必须能被关闭或绕过，不能让半成品成为唯一控制路径。

## 4. 禁止事项

以下操作默认禁止：

- 直接把上层阶跃命令喂给前馈或 DOB。
- 用直接阶跃结果证明前馈参数正确。
- 在多个文件复制默认参数。
- 用 positional pybind 参数长期传控制器配置。
- 修改仿真模型后不记录对指标的影响。
- 巡线、速度环、角速度环、DOB 同时调。
- 用 legacy 路径的结果证明主动路径正确。
- 发现量具污染后继续调参。

## 5. 文档更新规则

文档不要过度膨胀，但以下情况必须更新：

- 改了模块边界。
- 改了单位或参数含义。
- 改了主动控制路径。
- 新增了必须运行的验证命令。
- 发现原方案或旧计划有错误。

文档更新位置：

| 内容 | 文件 |
|---|---|
| 项目级规矩 | `项目维护协议.md` |
| Codex 入口规则 | `AGENTS.md` |
| Claude 入口规则 | `CLAUDE.md` |
| 控制理论和架构 | `控制器方案报告.md` |
| 当前执行计划 | `gpt计划.md` / `claude计划.md` |

## 6. 当前控制器改造总顺序

控制器继续改造时按这个顺序：

1. 文档和接口契约。
2. 参考整形层：`cmd -> ref/a/alpha`。
3. 纯前馈验证。
4. DOB。
5. 残差反馈。
6. 巡线。
7. 高速 sweep 和真实度开关。

在第 2 步完成前，不继续调巡线参数；在第 3 步可信前，不把闭环阶跃好看解释为前馈正确。
